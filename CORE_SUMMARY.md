## Core Summary — AI Learning Assistant

A document intelligence platform built across four independently deployed microservices: a React SPA, a Spring Boot REST API (backend), a Spring Boot RabbitMQ consumer (worker), and a Python FastAPI service that wraps Gemini. The core of the system is a message-driven pipeline: the backend publishes a `DocumentProcessingMessage` to a RabbitMQ queue when a PDF is uploaded, the worker consumes it and calls the AI service to generate a summary, flashcards, and a quiz, then publishes the results back on a second queue — the backend consumes that and writes everything to the database. The **single writer principle** is strictly enforced: the worker never touches the database, which eliminates dual-write consistency problems and gives one service clear ownership of all writes. The **Dependency Inversion Principle** is applied at the architecture level: the worker calls the AI service's HTTP interface rather than calling Gemini directly, meaning the LLM can be swapped by changing only one service. Resilience is built into the worker via 3-attempt exponential backoff retry and a dead-letter queue that captures messages after all retries are exhausted. Security uses a zero-trust JWT model: the backend validates Supabase ES256 tokens via JWKS (the public key only — the signing secret is never held by the backend), and RLS policies in PostgreSQL enforce row-level data isolation between users. Rate limiting is Redis-backed Bucket4j with Lua-script atomic operations, making it correct under horizontal scaling. A `correlationId` is generated at upload time, embedded in the queue message, and injected into MDC by every service that handles it — giving end-to-end log tracing across the pipeline without a distributed tracing framework.
